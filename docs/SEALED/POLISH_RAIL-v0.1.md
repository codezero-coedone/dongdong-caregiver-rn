# POLISH_RAIL — UX 폴리싱 레일 (v0.1, SEALED)

목적: “프론트 감도/UX 폴리싱”을 감(주관)이 아니라 **레일(규격)**로 고정한다.  
원칙: **최소 diff + 증거(DBG rid) 기반 + 실기기 기준**으로만 진행한다.

적용: RN 2종(Caregiver/Guardian) 공통. 우선은 **T7(매칭) → T8(간병일지 CRUD)** 폴리싱에 집중한다.

---

## 0) 폴리싱 원칙(강제)

- 폴리싱은 기능/계약을 바꾸지 않는다(이탈 포인트만 제거).
- 1폴리싱=1커밋(원인/증거/변경/DoD를 한 덩어리로).
- 증거 우선: DBG에서 `[rid=...]` 1줄 확보 후 수정.
- 실기기 기준: 에뮬레이터 의존 0.
- Korean UX: 범용 CTA “다음” 금지(상태 기반: 완료/확인/계속/재시도).

---

## 1) 폴리싱 6대 축(모든 화면 공통)

1) Touch/Response: 탭 1회=동작 1회(중복 호출/중복 네비게이션 0), 무반응 0  
2) Loading/Disable: 비동기 중 disable+로딩, 무한대기 0  
3) Empty/Zero: 데이터 0개여도 막힘 0, 안내+CTA 제공  
4) Error Message: 유저 문구는 짧게, 기술/클래스명/스택 노출 0(상세는 DBG)  
5) Keyboard/Form: 입력/탭 씹힘 0, 키보드로 CTA 가림 0  
6) Re-entry/Persistence: 재진입/재시작/로그아웃 후에도 상태 일관

---

## 2) 폴리싱 티켓(한 장 규격)

- Scope(화면/구간): 예) `T8-3 간병일지 작성`  
- Symptom(유저 관점): 예) “저장 눌러도 무반응/중복 저장”  
- Evidence: DBG `[rid=...] METHOD /path -> status` 1줄 (+ 필요 시 서버 rid 1줄)  
- Fix: 6대 축 중 해당 축만 최소 수정  
- DoD: “탭 1회=동작 1회”, “빈상태 막힘 0”, “재진입 동일”

---

## 3) T7~T8 우선 폴리싱 체크

- T7: `/my/matches` 0개여도 막힘 0(안내+CTA, DEV 레일은 DEVTOOLS=1에서만)  
- T8: `/journals` 조회/빈상태/작성/수정(409 포함)/재진입까지 한 사이클 완주

---

## 4) 검증 루프(고정)

실기기 재현 → DBG rid 확보 → (필요 시) 서버 rid 매칭 → 최소 수정 → 실기기 재검증 → 커밋

---

## 5) 금지

- 폴리싱 커밋에서 상태머신/라우팅/계약 구조 변경 금지
- DEV 도구가 프로덕션에서 켜지는 경로 추가 금지(DEVTOOLS gate 유지)
